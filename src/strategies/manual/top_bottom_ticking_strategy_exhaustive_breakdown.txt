TOP_BOTTOM_TICKING STRATEGY - EXHAUSTIVE DETAILED BREAKDOWN

Document purpose

This document is a deep, plain-text explanation of the top_bottom_ticking model/strategy based on the uploaded files ict_top_bottom_ticking.py and top_bottom_ticking_shared.py.

This write-up aims to leave as little as possible to inference. It explains what the strategy is trying to do, how its data is prepared, exactly how a setup is detected, how a pending setup becomes a trade, how stops and targets are calculated, how partials and breakeven moves work, how session controls work, how the backtest harness wraps the raw strategy, how symbol-specific tuning works, and how prop-firm rules are attached through the shared harness.

High-level summary

At its core, top_bottom_ticking is a short-term reversal model built around the idea of a price sweep into an important external liquidity level followed by rejection and a retracement entry.

In plain terms, price runs above an important prior high or below an important prior low, then closes back through that external level, suggesting rejection. The strategy defines the sweep candle as a zone, calculates the midpoint of that candle zone, called CE or consequent encroachment, and instead of entering immediately, waits for price to retrace back to that CE area. Depending on configuration, it may also require a simple change-of-structure style confirmation. When the retrace and confirmation happen, it enters with a stop just beyond the sweep zone, then manages the trade with three R-based targets and staged partial exits.

The strategy has two main styles: type2_baseline and type1_sniper.

The shared harness layers in symbol-specific stop rules, symbol-specific point values, prop-firm pre-entry guard logic, and reporting/CSV output.

File: ict_top_bottom_ticking.py

1. Imports and structural role

The core file imports numpy, pandas, Strategy from backtesting.py, and dataclass for pending setup state.

This file is the raw strategy engine. It is not the reporting harness. It is not the prop-firm wrapper. It is the actual trade idea implementation.

2. Global constants

The file defines:

* TICK_SIZE = 0.25
* FORCE_FLAT_HOUR_ET = 16
* FORCE_FLAT_MINUTE_ET = 50
* GLOBEX_REOPEN_HOUR_ET = 18

Meaning:

* the default tick size in the raw file is 0.25
* there is a force-flat cutoff at 16:50 ET
* Globex reopen reference is 18:00 ET

The shared harness later overrides symbol-specific behavior, including tick size and stop rules, but the raw base file starts from these defaults.

3. PendingSetup dataclass

The strategy uses a dataclass called PendingSetup with these fields:

* direction
* created_bar
* expiry_bar
* entry_ce
* zone_high
* zone_low
* stop_price
* target1
* target2
* target3
* external_level
* setup_type
* entry_variant
* internal_sweep

This object is central to the model. The strategy does not immediately trade the sweep candle. Instead, it stores the proposed trade as a pending setup and waits for the retrace entry.

Field meanings:

* direction: short or long
* created_bar: bar index where the pending setup was created
* expiry_bar: last bar index on which the setup is valid
* entry_ce: midpoint of the entry zone, the intended retracement entry reference
* zone_high: top of the sweep/rejection zone
* zone_low: bottom of the sweep/rejection zone
* stop_price: stop loss level
* target1/2/3: first, second, and third targets
* external_level: external buyside or sellside level that got swept
* setup_type: text label describing the setup
* entry_variant: whether the entry is CE only or CE plus structure confirmation
* internal_sweep: whether an internal sweep condition was present when the setup formed

4. Time conversion helpers

The strategy builds ET-based context from the original index.

Helper 1, _to_et(idx):

* takes the dataframe index
* if timezone-naive, assumes UTC
* converts it to America/New_York

This matters because the strategy’s session logic and Asia-window logic are defined in ET.

Helper 2, _session_date_for_et(ts):

* futures-style session labeling
* if ET hour is 18:00 or later, that bar belongs to the next session date

So 18:15 ET on a day is labeled as the next futures session date, not the prior calendar day. This matches CME-style session handling rather than midnight-to-midnight calendar logic.

5. build_model_frame(df)

This is one of the most important functions in the strategy.

Its purpose is to take raw OHLCV bars and add all derived context fields the model needs, create external liquidity levels, create 15-minute context markers, and create simple structure/sweep proxies.

5.1 Column normalization

* open -> Open
* high -> High
* low -> Low
* close -> Close
* volume -> Volume

5.2 ET time fields
It converts the index into ET and adds:

* et_time
* et_date
* et_hour
* et_minute
* session_date

These are used for session logic, Asia-window grouping, force-flat timing, and futures session labeling.

5.3 Prior session summaries
The strategy groups by session_date and computes the session high and low, then shifts them by one session and renames them to prior_session_high and prior_session_low.

This gives the current session access to the previous session’s high and low, which are treated as major external liquidity candidates.

5.4 Asia window proxy
The strategy defines an Asia window proxy using rows where et_hour >= 18. It groups these rows by session_date and computes asia_high and asia_low.

This is a simplified Asian-session liquidity reference for the current futures session.

5.5 External liquidity levels
The strategy creates:

* external_buyside = max(prior_session_high, asia_high)
* external_sellside = min(prior_session_low, asia_low)

This means on the upside it uses the higher of the prior session high and Asia high, and on the downside it uses the lower of the prior session low and Asia low.

5.6 15-minute context
The model resamples the 5-minute stream into 15-minute bars and creates Open, High, Low, Close, and Volume. It then computes:

* range = High - Low
* body = abs(Close - Open)
* bull = Close > Open
* bear = Close < Open

It identifies 15-minute swing points using a centered rolling window of 5 bars:

* swing_high_15 is true if the current bar’s high is the max in that 5-bar window
* swing_low_15 is true if the current bar’s low is the min in that 5-bar window

Then it merges the latest backward 15-minute context into the 5-minute frame using merge_asof, adding:

* ctx_high_15
* ctx_low_15
* ctx_close_15
* swing_high_15
* swing_low_15

These are context fields prepared for filtering or expansion, though the visible entry logic does not rely heavily on them.

5.7 Execution context fields
The model computes:

* atr14: average of High - Low over the last 14 bars, a simplified ATR proxy
* recent_high_6: rolling 6-bar highest high, shifted by 1
* recent_low_6: rolling 6-bar lowest low, shifted by 1
* recent_high_12: rolling 12-bar highest high, shifted by 1
* recent_low_12: rolling 12-bar lowest low, shifted by 1

These are used as short-term structure references.

5.8 Sweep conditions
The model defines sweep_short as true when:

* external_buyside exists
* current High >= external_buyside + one tick
* current Close <= external_buyside

Interpretation:
Price traded above the external buyside liquidity level by at least one tick, but then closed back at or below that level. That is treated as a bearish sweep and rejection.

The model defines sweep_long as true when:

* external_sellside exists
* current Low <= external_sellside - one tick
* current Close >= external_sellside

Interpretation:
Price traded below the external sellside liquidity level by at least one tick, but then closed back at or above that level. That is treated as a bullish sweep and rejection.

This is the heart of the model. It is not just a touch of the level. It requires a breach and then a reclaim/rejection back through it on close.

5.9 Internal sweep proxies

* internal_sweep_short = High >= recent_high_6 + one tick
* internal_sweep_long = Low <= recent_low_6 - one tick

These are smaller internal liquidity sweeps inside recent local structure. In the base strategy, these are only enforced if require_internal_sweep_filter = True.

5.10 Change-of-structure proxies

* cos_short = Close < recent_low_6
* cos_long = Close > recent_high_6

These are simple structure-break proxies. They are intentionally lightweight rather than full market-structure engines.

10. Setup arming logic

10.1 _arm_short_from_sweep(row, i)
A bearish setup starts when a valid bearish sweep has been detected.

Internal filter:

* if require_internal_sweep_filter is True and internal_sweep_short is False, reject the setup and increment blocked_internal_filter.

Zone construction:

* zone_high = sweep candle High
* zone_low = min(Open, Close)
* entry_ce = midpoint of zone_high and zone_low
* stop = zone_high + one tick
* risk = stop - entry_ce

Risk filter:

* reject if risk < min_stop_points
* reject if risk > max_stop_points

If valid, it arms a PendingSetup with:

* short direction
* creation and expiry bars
* entry CE
* zone boundaries
* stop
* targets at 1R, 2.25R, and 4.25R below entry CE
* external level reference
* setup type label
* entry variant label
* internal_sweep flag

10.2 _arm_long_from_sweep(row, i)
Mirror image of the short logic:

* zone_low = sweep candle Low
* zone_high = max(Open, Close)
* entry_ce = midpoint
* stop = zone_low - one tick
* risk = entry_ce - stop
* reject if risk is outside allowed stop range
* if valid, arm a long PendingSetup with 1R, 2.25R, and 4.25R upside targets

11. Pending setup readiness rules

11.1 _pending_short_ready(row)
For a pending short to trigger:

* current High must reach entry_ce minus tolerance
* if COS confirmation is required, cos_short must be true and Close must be below entry_ce
* if COS confirmation is not required, Close <= entry_ce is enough

11.2 _pending_long_ready(row)
For a pending long to trigger:

* current Low must reach entry_ce plus tolerance
* if COS confirmation is required, cos_long must be true and Close must be above entry_ce
* if COS confirmation is not required, Close >= entry_ce is enough

12. Actual entry logic

12.1 _enter_short(row, i)
Executed once pending short is ready.

* entry = min(current Close, pending.entry_ce)
* stop = pending.stop_price
* risk = stop - entry
* if risk <= 0, cancel

Targets are then re-anchored to the actual executed entry:

* target1 = entry - 1R
* target2 = entry - 2.25R
* target3 = entry - 4.25R

Sanity check:

* target3 < entry < stop

Then:

* active_risk is stored
* a sell order is placed with size fixed_size, stop loss, and final TP at target3
* partial flags reset
* trade metadata recorded
* entry_short counter incremented
* pending setup cleared

12.2 _enter_long(row, i)
Mirror image:

* entry = max(current Close, pending.entry_ce)
* stop = pending.stop_price
* risk = entry - stop
* targets are re-anchored to actual entry
* sanity check ensures stop < entry < target3
* then it buys with stop and final TP at target3

13. Trade management

13.1 General structure
Trade management runs only if:

* there is an active trade
* there is an active position
* active_risk is finite

13.2 Long trade management
For longs:

* r_now = (current price - entry_price) / active_risk

At 1R:

* close 40 percent of the position
* mark partial1_taken

After target1:

* if enabled, move stop to breakeven or better
* mark be_moved

At 2.25R:

* close two-thirds of the remaining size
* mark partial2_taken

With fixed size 5:

* first partial takes off 2 units
* 3 remain
* second partial removes 2 of those 3
* 1 unit remains as the final runner

13.3 Short trade management
Mirror image:

* r_now = (entry_price - current price) / active_risk
* first partial at 1R
* stop moved to breakeven or better
* second partial at 2.25R
* last unit remains for target3

This structure locks gains early, reduces risk quickly, and still leaves tail exposure for bigger moves.

14. next() - the full bar-by-bar state machine

Execution order each bar:

1. log newly closed trades
2. warmup filter; do nothing before 150 bars
3. read current derived row
4. manage open trade
5. force-flat if needed
6. expire stale pending setups
7. if a position is open, stop processing
8. if after force-flat cutoff, clear pending and stop
9. if a short is pending and ready, enter short
10. if a long is pending and ready, enter long
11. if any setup is already pending, do not arm another one
12. if flat and no pending setup, detect new sweep setups
13. if a short sweep is found, arm a short setup
14. if a long sweep is found, arm a long setup

The model is therefore a single-position, single-pending-setup state machine.

* entry proceeds

This is true pre-entry filtering, not post-trade filtering.

PART 3 - THE STRATEGY IDEA IN HUMAN TERMS

26. Conceptual setup

This is a liquidity sweep reversal strategy.

For shorts:

* identify an upper external liquidity level
* price trades above it
* price closes back below it
* this suggests a failed breakout or rejection
* define the rejection candle’s upper portion as the zone
* compute the midpoint of that zone
* wait for retracement to that midpoint
* optionally require bearish structure confirmation
* short from that retracement/confirmation
* stop above the zone
* take profits in stages

For longs:

* identify a lower external liquidity level
* price trades below it
* price closes back above it
* define bullish rejection zone
* compute midpoint
* wait for retracement
* optionally require bullish structure confirmation
* buy
* stop below the zone
* scale out at fixed R multiples

27. Why CE midpoint is used

The model does not enter immediately on the sweep candle close. It waits for a retracement into the midpoint of the rejection zone.

Likely reasons:

* better entry price
* better R multiple
* less chasing
* structurally consistent with rejection-block / CE execution logic

28. Why COS confirmation matters

If COS confirmation is required:

* the strategy is stricter
* the retrace alone is not enough
* price must also show a simple structure break

If COS confirmation is not required:

* more entries can fire
* the model is more aggressive and permissive

29. Why stop-size filters matter

The strategy rejects setups whose stop distance is too small or too large.

Too small:

* too noise-sensitive
* easy to stop out
* unrealistic in some markets

Too large:

* poor efficiency
* weak R profile
* too much exposure for the setup style

30. Why one-position-at-a-time matters

The model does not pyramid and does not hold multiple overlapping trades. It is:

* single setup
* single open position
* single pending setup

This keeps state simpler and makes prop-guard attachment easier.

31. Why the partial structure looks like this

The sequence is:

* 40 percent off at 1R
* stop to breakeven
* two-thirds of remainder off at 2.25R
* final runner to 4.25R

This balances:

* early monetization
* fast risk reduction
* tail participation if the move expands

PART 4 - DIFFERENCE BETWEEN BASELINE AND SNIPER

32. Baseline behavior

Baseline is generally the stricter version.
Common traits:

* stricter stop range
* tighter touch tolerance
* COS confirmation required
* fewer but more filtered entries

33. Sniper behavior

The configured sniper is looser in the current implementation.
Common traits:

* COS confirmation disabled
* wider touch tolerance
* slightly wider stop allowance
* often more entries

So despite the name, the current sniper often behaves as the higher-throughput variant.

PART 5 - IMPORTANT LIMITATIONS / ASSUMPTIONS

34. Simplified structure model
    The COS and internal sweep logic are simple proxies, not a full market-structure engine.
35. Asia session proxy
    The Asia session proxy is simplified to bars with ET hour >= 18.
36. ATR proxy is simplified
    atr14 is just average bar range, not classical ATR.
37. One pending setup only
    If a setup is already pending, new setups are ignored until it triggers, expires, or is cleared.
38. One open trade only
    No overlapping positions.
39. Force-flat behavior
    Positions are forcibly closed after cutoff and no new setups are allowed after that point.
40. Shared file depends on external prop profile/guard files
    This breakdown explains how the harness uses them, but not their entire internals, because those extra files were not part of the inspected strategy pair here.

Still, one thing is clear:

* prop rules are attached as a wrapper
* they do not rewrite the raw model
* they are enforced before entry

PART 6 - BOTTOM LINE

41. One-sentence strategy definition
    It is a short-term futures reversal model that looks for sweeps of external liquidity, defines the rejection candle as an entry zone, waits for a midpoint retracement with optional structure confirmation, and manages the trade through multi-stage R-based exits.
42. One-sentence harness definition
    It turns the raw strategy into a multi-symbol, multi-variant, point-value-aware, prop-profile-compatible backtest/forward-test framework with normalized reporting and pre-entry trade blocking.
43. Practical loop summary
    For each bar:

* update context
* manage any open trade
* force-flat if required
* expire stale setups
* if already in a trade, do nothing else
* if a setup is pending, wait for retrace/confirmation
* if nothing is pending, detect a fresh external sweep
* arm a pending setup
* if retrace comes back and rules pass, enter
* manage partials and stop
* record the trade when it closes

* In your setup this can come from Databento, TradingView bars, or a fallback fetcher path.

Layer 5 - Prop-firm rules layer

* This applies account/risk constraints separately from model logic.
* The raw strategy does not inherently know Apex rules.
* The prop guard layer adds that behavior before entry.

Layer 6 - Execution routing layer

* This takes a valid entry/action and sends it to your execution bridge.
* In your current operating path, this goes through PickMyTrade.

Layer 7 - Broker / prop account layer

* This is where the actual account exists.
* In your case this is your Apex PA account routed through Tradovate infrastructure.

Layer 8 - Monitoring / state / logs

* This stores runtime state, event logs, and health signals.
* It allows the runner to know what it already processed and what happened on prior cycles.

48. How the strategy logic flows in backtesting mode

In backtest mode, the flow is generally:

1. Data is loaded for a symbol.
2. The raw top_bottom_ticking strategy class is selected.
3. The shared harness modifies the class according to:
    * symbol
    * variant
    * prop profile
4. The backtest engine runs the strategy bar by bar.
5. The raw strategy detects sweeps, arms pending setups, enters, manages exits, and logs metadata.
6. The shared harness converts raw trade information into symbol-aware dollar PnL.
7. CSV summaries are produced.

In this mode, nothing is sent to a broker. It is all simulated.

49. How the strategy logic flows in deployed / forward-test mode

In deployed mode, the flow is different.

The deployed top_bottom_ticking wrapper does not simply replay a whole year.
Instead it runs in a live-style cycle.

A single cycle looks like this:

1. The deployed app wakes up.
2. It checks the current ET time and session status.
3. If market is closed, it usually idles and logs a heartbeat.
4. If entries are allowed, it calls generate_live_actions().
5. The deployed live_model wrapper scans each configured symbol.
6. For each symbol it runs each configured variant.
7. It loads recent bars for that symbol.
8. It runs the raw strategy over recent history using the backtesting engine as a signal extractor.
9. It inspects the latest bars and latest trade log metadata.
10. It extracts only fresh entries that fall within the configured recent lookback window.
11. It converts those into normalized live actions.
12. The app runner checks state so it does not resend already-seen signals.
13. If live orders are disabled, it simulates processing only.
14. If live orders are enabled, it routes the action into the execution layer.
15. Monitoring and state are updated.
16. The process sleeps until the next cycle.

So the deployed model is essentially using the same raw trading logic as a signal engine, but wrapping it in a repeating operational loop.

50. How data gets into the strategy

There are several possible data paths.

Path A - Databento

* Databento is your preferred direct market data source.
* It provides CME futures market data.
* The deployed live model can warm up historical bars and then consume live/near-live bars from Databento.

Path B - TradingView bars

* Your system can also read TradingView-exported or TradingView-fed bar files.
* These are stored in your tradingview_bars folder.
* The deployed model can use these directly or fall back to them when Databento is unavailable or incomplete.

Path C - Fallback fetcher

* There is also a fallback data fetcher path through your local repo utilities.
* This is generally a lower-priority safety path.

Operationally, the typical preference is:

1. Databento
2. TradingView fallback
3. fetcher fallback
4. Why multiple data paths exist

You use multiple data paths because each serves a different operational purpose.

Databento:

* strongest direct market-data route
* better for robust live operation
* more aligned with institutional futures feed usage

TradingView bars:

* useful because your broader bot stack already interacts with TradingView
* good fallback when direct feed handling is still being stabilized
* also useful if live Databento buffers are incomplete for some symbols at startup

Fetcher fallback:

* final safety net

This layered data design increases runtime resilience.

52. How top_bottom_ticking live_model uses the raw strategy

The deployed live_model for top_bottom_ticking does not reimplement the strategy rules by hand.
Instead it imports them.

Broadly it does this:

* imports the shared manual module
* reads the instrument definitions
* reads the variant class map
* selects the relevant class per symbol and per variant
* loads recent bars
* runs a short backtesting pass on the recent data
* extracts only fresh entry opportunities from the raw strategy’s metadata output

This is very important conceptually.

The deployed file is not a second independent strategy.
It is a runtime wrapper around the same core strategy logic.

53. Why the deployed model still uses backtesting.py internally

Even in deployed scanning mode, the system uses the backtesting engine as a convenient way to run the strategy class over recent bars.

Why this is useful:

* it reuses the exact same entry/exit logic as backtests
* it reduces duplication
* it keeps live-style signal generation closer to backtest behavior
* it avoids rewriting the whole strategy state machine a second time for deployment

So in deployed mode, the backtesting engine is being used less as a portfolio backtester and more as a deterministic strategy runner over recent bars.

54. How fresh entries are extracted

After the raw strategy is run over recent data, the deployed wrapper does not blindly turn every historical trade into a live signal.

Instead it:

* looks at the recent bar window
* checks trade metadata/log entries
* finds only entries whose timestamps are within the latest lookback bars
* converts those into actionable signals

This is how it tries to avoid replaying stale historical trades.

56. Why signal_id matters

signal_id prevents duplicate processing.

The deployed app keeps a state file with seen signal IDs.
If the same signal appears again on the next cycle, it will not be resent.

This is critical in any looping live system. Without it, the model could repeatedly send the same trade every cycle.

57. Session and market-time handling

The deployed runner is ET-aware.
It uses:

* force-flat time
* Globex reopen time
* weekday/weekend logic

This means the runner knows whether it is:

* in normal session
* in force-flat window
* in weekend-closed state
* in Sunday pre-open state

When market is closed:

* it can avoid entry generation
* it can emit a heartbeat rather than look frozen
* it can sleep longer between cycles

58. Force-flat logic in the full system

There are two related force-flat ideas:

Inside the raw strategy:

* if a position is open after the cutoff, it closes it
* no new setups are allowed after the cutoff

Inside the deployed runner:

* if force-flat behavior is enabled, the runtime can also simulate or perform account-wide flattening behavior based on current session state

So force-flat is both:

* a strategy behavior
* a runtime safety behavior

59. Prop-firm rules in the full system

This is one of the most important architecture decisions.

The raw strategy does not intrinsically know Apex rules.
Instead, prop-firm rules are connected outside the raw model through the prop guard layer.

This separation is deliberate and important because:

* the same strategy can be used with different prop firms
* the model logic remains stable
* account rule logic can be swapped or toggled through a profile

That means the architecture is:

raw model logic
+
prop-firm rule wrapper
+
runtime execution wrapper

not:

raw model logic hardcoded specifically for Apex only

60. How a prop profile affects actual behavior

When a prop profile is active, the system may block trades before entry based on things such as:

* daily loss conditions
* consecutive losses
* max trades
* trailing drawdown or similar thresholds depending on profile implementation

In the shared harness, this happens before entry.
That means if a trade is disallowed:

* the trade is never opened
* it is not filtered out after the fact
* the strategy’s pending setup is cleared
* the block reason is counted for reporting

This is the cleanest way to attach prop rules.

61. Why this separation matters for future firms

Because the raw strategy and prop rules are separated, you can later use:

* Apex
* Topstep
* other eval/funded rule sets

without rewriting the core entry logic.

You mainly change:

* the selected prop profile
* the rules file/profile definition

not the raw model.

62. How execution is routed in your current setup

Your current intended route is:

This means the bot itself is not directly making Tradovate broker calls as the primary route in the live deployment path.

Instead:

This makes PickMyTrade your main execution bridge layer.

PickMyTrade acts as the operational bridge between your signal engine and your prop account execution path.

Its role is to:

* receive order instructions from your bot
* map them to the destination account
* forward them to the Tradovate/Apex side

So in your live workflow, PickMyTrade is effectively the central execution connector.

65. What happens when both models run together

When you run both:

* ict_fractal
* top_bottom_ticking

through run_models.py, the launcher starts both deployed apps.

Each model then:

* loads its own config
* runs its own live loop
* reads its own data path
* generates its own actions
* uses the shared or model-specific execution wrapper
* targets the same prop profile / account if configured that way

This means both models can conceptually trade the same account.

That has important consequences:

* they share account balance
* they share prop-firm risk budget
* they can overlap in exposure
* they can independently generate signals in the same time window

66. Why dual-model deployment requires caution

If two models trade the same Apex PA account, risk control becomes more important.

Even if each model individually looks fine, together they can:

* increase trade count
* increase drawdown speed
* overlap on correlated positions
* consume the same loss budget

So dual-model deployment is possible, but should usually be staged carefully:

* both models in no-live-orders mode first
* then one live and one simulated
* then both live at very small size

71. End-to-end workflow from data to account

A practical end-to-end live workflow looks like this:

1. You start run_models.py.
2. The launcher reads the registry and starts top_bottom_ticking app.py.
3. app.py loads config and enters a live loop.
4. app.py checks whether entries are allowed at the current ET session time.
5. If allowed, app.py calls live_model.generate_live_actions().
6. live_model loads recent bars from Databento, TradingView, or fallback path.
7. live_model imports the shared top_bottom_ticking strategy classes.
8. live_model runs the selected symbol and variant combinations through recent bars.
9. The raw strategy identifies fresh setups, pending entries, and actual entry conditions.
10. The wrapper extracts only fresh entries from the latest recent window.
11. The app checks whether that signal was already processed.
12. If prop/runtime rules allow it and live orders are enabled, the action is passed to execution.py.
13. execution.py routes the action into the execution bridge layer.
14. The bridge route is PickMyTrade.
15. PickMyTrade forwards the execution intent to the Tradovate/Apex side.
16. The Apex PA account receives the order on the broker side.
17. Monitoring/state files are updated.
18. The loop sleeps until the next cycle.
19. End-to-end workflow in no-live-orders mode

When you run with no-live-orders:

* all the above can still happen up to signal generation and action processing
* but the final actual live order send is blocked/simulated

This is extremely useful because it allows you to validate:

* startup
* imports
* session behavior
* signal generation
* multi-model compatibility
* runtime logs

without risking live fills.

No architecture is perfect. Important practical risks include:

B. Multi-model account contention

C. Duplicate or stale signal risk

D. Runtime import/config mismatch risk

* especially when new deployed models are added

E. Execution bridge dependency

* if PickMyTrade has issues, execution can fail even if the model is correct

77. Final one-paragraph summary of the full workflow

The full top_bottom_ticking workflow starts with recent futures market data coming in from Databento or TradingView, after which the deployed live wrapper loads the raw strategy classes from the manual strategy files and runs them over recent bars using the backtesting engine as a signal extractor. The raw strategy looks for sweeps of external liquidity, arms pending setups, waits for midpoint retrace conditions, and identifies valid entries. The deployed wrapper extracts only the fresh entry opportunities, normalizes them into action objects, checks state so they are not duplicated, applies session/runtime behavior and, where relevant, prop-profile-based pre-entry guards, then routes the final action into the execution bridge. In your current architecture, that bridge goes to PickMyTrade, which then routes execution toward Tradovate and ultimately your Apex PA account. Around all of this sit config files, monitoring logs, state files, the model registry, and the run_models launcher, which together form the operational framework for running the strategy safely in backtest, forward test, or live mode.
